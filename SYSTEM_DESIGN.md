# 评审系统重设计规划

## 定位

系统从单一虚拟项目演示，调整为面向多个科研项目的评审工作台。当前阶段先不建设正式鉴权/RBAC，优先完成项目提交、项目切换、三阶段评审清单生成、评审状态持久化和本体/知识库/LLM 联调链路。

登录入口先拆分为评审专家、申报方和管理员三类演示入口。入口会生成对应前端会话角色，用于驱动现有界面权限提示和操作可见性，但不代表正式后端鉴权。

## 核心对象

- 项目：申报方提交的评审对象，包含项目名称、申报单位、预算、周期、领域、联系人、结构化申报材料和附件清单。
- 阶段：立项评审、中期检查、结题验收。
- 评审项：每个阶段的结构化检查清单，包含状态、专家评分、只读 AI 评分、本体校验、LLM 建议和历史记录。
- 证据：静态知识库命中文档、结构化申报材料、上传附件解析文本、推理链片段、本体路径和材料缺口。
- 活动：项目提交、查看依据、提交结论、要求补材料等工作台动态。

## 当前架构

```text
React 工作台
  |-- 项目中心：独立页面，负责项目列表、项目切换、提交新项目
  |-- 评审工作区：阶段概览、评审项、评分意见、历史
  |-- 证据区：推理链、本体路径、知识库引用
  `-- 助手区：聊天、辅助意见生成、引用资料

Node HTTP API
  |-- /api/projects：项目列表与提交
  |-- /api/projects/current：切换当前项目
  |-- /api/app-state：当前项目 + 当前阶段工作台状态
  |-- /api/review-stage：阶段流转
  |-- /api/review-items/:id：评审项更新、历史、依据
  |-- /api/knowledge-base：知识库检索
  |-- /api/llm/complete：辅助意见生成
  `-- /api/chat：评审助手

Provider 层
  |-- Knowledge Base Provider
  |-- Ontology Validation Provider
  |-- LLM Provider
  `-- Project Evidence Provider

Local Runtime Store
  `-- projectStates[projectId].stageStates[stage]
```

## 数据结构调整

运行态从原来的单项目形态：

```json
{
  "currentStage": "proposal",
  "stageStates": {}
}
```

升级为多项目形态：

```json
{
  "currentProjectId": "project-001",
  "projectOrder": ["project-001"],
  "projectStates": {
    "project-001": {
      "project": {},
      "currentStage": "proposal",
      "submittedAt": "2026-04-11T00:00:00.000Z",
      "source": "demo",
      "stageStates": {
        "proposal": {},
        "midterm": {},
        "final": {}
      }
    }
  }
}
```

旧运行态会自动迁移为一个 `demo` 项目，不需要手工清理。

## 项目提交流程

1. 用户进入独立“项目中心”页面。
2. 填写项目名称、申报单位。领域、联系人、联系方式、预算和周期可以后补。
3. 填写核心申报材料：项目摘要、研究目标、技术路线。
4. 可补充创新点说明、阶段里程碑、团队与分工、预算拆分说明、预期成果、风险预案、伦理与合规说明、附件清单说明，并通过上传入口选择申报书、预算表、合作协议、证明材料等支撑文件。
5. 后端创建项目 ID，保存文件对象，并将结构化材料与 PDF / Word / Excel 附件解析结果写入项目证据索引。
6. 后端使用当前评审模板生成三阶段评审清单。
7. 新项目自动成为当前项目，并进入立项评审阶段。
8. 工作台加载该项目独立的评审项、活动流、历史、聊天上下文和项目证据索引。

新提交项目不会继承演示项目的评分、意见和争议状态，只继承评审清单结构。

## 阶段流转

- 当前项目内独立计算阶段状态。
- 当前阶段默认进入 `in_progress`。
- 后续阶段如果只有未处理模板项，会显示为 `not_started`。
- 只有前置阶段完成，才允许进入后续阶段。
- 待补材料、争议项、未完成项都会阻塞后续阶段。

## 本阶段暂不处理

- 正式账号体系
- 后端 RBAC 权限
- 多人并发编辑锁
- 数据库迁移
- 生产级对象存储适配器、在线预览、图片 OCR、PPT/压缩包解析和补材料闭环

这些能力后续应该进入生产化路线，但不阻塞当前多项目系统形态。

## 下一步建议

1. 项目列表筛选：支持按状态、阶段、申报单位、提交时间筛选。
2. 项目材料：把当前本地开发 Project Evidence Provider 替换为数据库/对象存储实现，并增加在线预览和补材料记录。
3. 评审模板管理：把三阶段评审项从代码模板迁移到可配置数据。
4. 评审结论页：生成项目级阶段报告，汇总评分、证据、争议项和补材料项。
5. 数据库：把 `runtime-state.json` 替换为项目、阶段、评审项、历史和活动表。
